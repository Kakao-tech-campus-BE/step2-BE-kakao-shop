## **과제 수행 내용**

### **1번**

**1. 전체 상품 목록 조회**

HTTP 요청 메서드와 URL : GET http://localhost:8080/store/products
기존의 /products는 판매를 위한 상품 목록 조회인지 장바구니의 상품 목록인지 등을 알기가 힘들다.
따라서 /store를 추가하여 스토어에 판매하고 있는 상품 목록임을 api만 보고 알 수 있게 한다.
</br>

**2. 개별 상품 목록 조회**

HTTP 요청 메서드와 URL : GET http://localhost:8080/store/products/{id}
마찬가지로 /store를 추가하여 판매 상품 목록 조회임을 알 수 있게 한다.
</br>

**3. 이메일 중복 체크**

HTTP 요청 메서드와 URL : POST http://localhost:8080/join/check_email
기존의 /check URL은 어떤 것을 체크하는지 알기가 힘들다.
따라서 회원가입 단계에서 이메일의 유효성을 체크하는 기능임을 api를 보고 유추할 수 있게 한다.
</br>

**4. 회원가입**

HTTP 요청 메서드와 URL : POST http://localhost:8080/join
기존의 URL, 요청 메서드가 회원가입 기능을 잘 표현한다고 생각하여 그대로 사용하였다.
</br>

**5. 로그인**

HTTP 요청 메서드와 URL : POST http://localhost:8080/login
기존의 URL, 요청 메서드가 로그인 기능을 잘 표현한다고 생각하여 그대로 사용하였다.
</br>

**6. 장바구니 담기**

HTTP 요청 메서드와 URL : PATCH http://localhost:8080/cart
장바구니의 전체 목록을 수정하는 것이 아닌 일부를 수정하므로 PATCH 요청 메서드로 cart의 일부 내용을 수정한다.
</br>

**7. 장바구니 조회**

HTTP 요청 메서드와 URL : GET http://localhost:8080/cart
장바구니는 여러 개 존재할 수 없으므로(product는 여러 개가 될 수 있지만) carts 대신 cart를 사용하였다.
</br>

**8. 주문하기**

HTTP 요청 메서드와 URL : PUT http://localhost:8080/cart
주문하기 전에 장바구니를 최종적으로 한번 더 수정할 수 있다.
이 때 장바구니에서 전체 목록을 확인할 수 있으므로 주문하기 전 장바구니 수정은 PUT으로 cart의 모든 내용을 교체하는 맞다고 생각하여 사용하였다.
대부분 주문할 때 품목이 몇 십개가 되지는 않으므로 모든 내용을 교체해도 성능 저하는 크게 없다고 생각하였다.
</br>

**9. 결제하기**

HTTP 요청 메서드와 URL : POST http://localhost:8080/orders
결제가 완료되면 주문 내역이 새로 생성된다.
이 때 POST 만으로도 주문을 저장했다는 의미를 확인할 수 있기 때문에 save는 제외하였다.
</br>

**10. 주문내역 확인하기**

HTTP 요청 메서드와 URL : GET http://localhost:8080/orders/{id}
기존의 URL, 요청 메서드가 주문내역 기능을 잘 표현한다고 생각하여 그대로 사용하였다.
</br>
</br>
